Run multiple Claude Code (CLI) subscriptions side by side, in parallel
terminals. Each profile has its own long-lived OAuth token and config dir.
Shared customisation (`settings.json`, `hooks/`, `agents/`, `skills/`,
`plugins/`, `commands/`, etc.) is symlinked from your main `~/.claude/`, so a
tweak in one place applies everywhere.

**Scope**: Claude Code in the terminal (`claude` command) only. Claude Desktop
is a separate problem with a different auth mechanism and not in scope for
v0.1.

## Why

Switching accounts in Claude Code is painful:

- The built-in `/login` swaps credentials in place — you can't run two
  accounts simultaneously.
- On macOS, credentials live in the global Keychain, so the obvious
  `CLAUDE_CONFIG_DIR=~/.claude-personal` trick **does not** isolate auth
  (Anthropic docs confirm: `CLAUDE_CONFIG_DIR` only redirects credentials on
  Linux/Windows).

The escape hatch is `CLAUDE_CODE_OAUTH_TOKEN` — a long-lived OAuth token
generated by `claude setup-token`, which overrides the Keychain entry per
[Claude Code's documented auth precedence][auth-precedence]. Set it per
shell, and each terminal authenticates as a different account.

`cca` is the minimum useful wrapper around that: one bash script. Required
on PATH: `claude`, `python3`, `curl`.

[auth-precedence]: https://code.claude.com/docs/en/iam#authentication-precedence

## How it works

```
~/.claude/                         ← "main" profile (your existing setup, auth via Keychain)
~/.claude-personal/                ← profile created by `cca add personal`
  ├── .cca-profile                 ← marker so cca only lists its own dirs
  ├── .cca-token                   ← long-lived OAuth token (mode 0600)
  ├── settings.json   -> ~/.claude/settings.json    (symlink)
  ├── hooks/          -> ~/.claude/hooks/           (symlink)
  ├── skills/         -> ~/.claude/skills/          (symlink)
  ├── plugins/        -> ~/.claude/plugins/         (symlink)
  ├── agents/         -> ~/.claude/agents/          (symlink)
  ├── commands/       -> ~/.claude/commands/        (symlink)
  └── ...
```

`cca switch personal` emits two exports:

```bash
export CLAUDE_CONFIG_DIR=$HOME/.claude-personal     # settings/skills isolation
export CLAUDE_CODE_OAUTH_TOKEN=<token>              # auth isolation (overrides Keychain)
```

`cca switch main` emits `unset` lines for both — falling back to your default
Keychain-based login.

## Install

```bash
git clone https://github.com/<owner>/cca.git
cd cca
./install.sh
```

Symlinks `cca` into `~/.local/bin/`. Add that to your `$PATH` if needed.

## Quick start

```bash
# 1. Create a profile
cca add personal

# 2. Generate a long-lived OAuth token for that account
cca login personal
# - Runs `claude setup-token` (opens browser, log in with personal account)
# - Prompts you to paste the printed token
# - Saves to ~/.claude-personal/.cca-token (mode 0600)

# 3. Switch in your current shell
eval "$(cca switch personal)"

# 4. Run claude — it now uses the personal account
claude

# 5. In a different terminal, run with main (work) account
claude     # CLAUDE_CODE_OAUTH_TOKEN unset → falls back to Keychain
```

### Make `cca switch` work without `eval`

A child process (cca) can't modify its parent's shell environment directly,
which is why the bare command emits `export` lines for you to `eval`. Install
a tiny wrapper function once and `cca switch personal` will just work:

```bash
cca init                  # interactive: detects $SHELL, shows you the wrapper, asks to append
exec $SHELL               # reload

# now this works directly, no eval needed:
cca switch personal
cca switch main
```

Flags:

- `cca init -y` — non-interactive install (skip the confirmation prompt)
- `cca init --print` — print the wrapper instead of installing (for inspection / piping)
- `cca init zsh` / `cca init bash` — force shell type if `$SHELL` is wrong

`cca init` is idempotent: a second run sees the marker line and exits without
re-appending.

## Commands

| Command                    | Description                                                                                  |
| -------------------------- | -------------------------------------------------------------------------------------------- |
| `cca ls`                   | List profiles, mark active, show auth state                                                  |
| `cca active`               | Print active profile name                                                                    |
| `cca add <name>`           | Create profile dir + symlink shared items                                                    |
| `cca login <name>`         | Run `claude setup-token`, save the printed token                                             |
| `cca switch <name>`        | Emit `eval`-friendly export lines                                                            |
| `cca rename <old> <new>`   | Rename a profile (alias: `cca mv`)                                                           |
| `cca rm <name>`            | Delete profile dir (confirms first)                                                          |
| `cca refresh <name>`       | Re-create symlinks (after adding new items to `~/.claude/`)                                  |
| `cca usage [name]`         | Activity heatmap (12-month) + summary card per profile, built from local session transcripts |
| `cca version` / `cca help` | Self-explanatory                                                                             |

## What gets shared vs kept per-profile

**Shared** (symlinked from `~/.claude/`):

- `settings.json`, `keybindings.json`, `mcp.json`
- `CLAUDE.md`
- `hooks/`, `agents/`, `commands/`, `skills/`, `plugins/`
- `statusline.sh`
- `carta/`, `rules/` (your own custom dirs)

Edit the `SHARED_ITEMS` array in `cca` to add/remove items.

**Per-profile** (never symlinked):

- `.cca-token` — the OAuth token (this is the whole point)
- `.credentials.json` (Linux/Windows) — if Claude Code writes one
- Everything in `~/.claude-<name>/` that isn't on the shared list

## Why `CLAUDE_CODE_OAUTH_TOKEN` and not `CLAUDE_CONFIG_DIR` alone

Per [Anthropic's docs](https://code.claude.com/docs/en/iam#credential-management):

> On macOS, credentials are stored in the encrypted macOS Keychain.
> If you've set the `CLAUDE_CONFIG_DIR` environment variable **on Linux or
> Windows**, the `.credentials.json` file lives under that directory instead.

So `CLAUDE_CONFIG_DIR` does **not** isolate auth on macOS. The Keychain entry
is global. The only way to override it per shell is the documented auth
precedence: `CLAUDE_CODE_OAUTH_TOKEN` (item #5 in the precedence list) beats
the Keychain entry (item #6).

`cca` uses both vars together:

- `CLAUDE_CONFIG_DIR` → isolates settings, skills, plugins, hooks, etc. via
  the symlinked profile dir
- `CLAUDE_CODE_OAUTH_TOKEN` → isolates **auth**

This works on macOS, Linux, and Windows.

## Caveats

- **OAuth tokens last ~1 year.** Re-run `cca login <name>` to refresh.
- **`--bare` mode does not read `CLAUDE_CODE_OAUTH_TOKEN`.** If you script
  with `claude --bare`, authenticate with `ANTHROPIC_API_KEY` instead.
- **Remote Control sessions** can't use `cca`-managed tokens (the docs say
  long-lived tokens are "scoped to inference only and cannot establish Remote
  Control sessions"). Use your main Keychain account for those.
- **Starting 2026-06-15**: `claude -p` (non-interactive) on subscription
  plans draws from a separate Agent SDK credit pool, per Anthropic.
  Interactive `claude` is unaffected.
- **Claude Desktop is out of scope.** This tool only manages CLI auth.

## Cross-machine sync (manual for now)

`cca` doesn't ship sync. To replicate your config to another machine:

```bash
cd ~/.claude
git init -b main
cat > .gitignore <<'EOF'
.credentials.json
.cca-token
.claude.json
projects/
sessions/
session-env/
tasks/
history.jsonl
file-history/
shell-snapshots/
backups/
paste-cache/
image-cache/
ide/
debug/
.ccassist/
.search-index/
plugins/cache/
plugins/marketplaces/
EOF
git add -A
git commit -m "initial claude config"
git remote add origin git@github.com:you/claude-config.git
git push -u origin main
```

On the other machine: `git clone` to `~/.claude/`, then per profile run
`cca add <name> && cca login <name>` (tokens stay per-machine; you don't
want them in git).

A future `cca sync push/pull` may automate this.
